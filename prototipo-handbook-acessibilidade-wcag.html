Como usar este handbook

O objetivo não é substituir a WCAG, a ABNT NBR 17225 ou uma auditoria completa. O objetivo é reduzir o tempo entre perceber um problema e aplicar uma correção técnica inicial.

Sintoma observado → Causa provável → Solução mínima → Como validar → Critério WCAG relacionado

Para desenvolvimento

Use antes de criar componentes, telas, formulários e fluxos interativos. A consulta deve começar pelo comportamento esperado do usuário, não pelo número do critério.

Para correção de bugs

Use quando houver um sintoma claro, como “não funciona com Tab”, “leitor de tela não anuncia o erro” ou “o botão não tem nome acessível”.

Para governança

Use como base para padronizar respostas técnicas, criar cards de conhecimento e orientar sustentação de sistemas.

A busca filtra os itens deste protótipo sem recarregar a página.

Índice por sintomas

  • Não consigo acessar um componente usando Tab
  • O foco visual desaparece ou é difícil de perceber
  • O leitor de tela não informa o tipo ou o nome do controle
  • Campo de formulário é anunciado sem rótulo
  • Mensagem de erro aparece visualmente, mas não é anunciada
  • Texto ou ícone tem baixo contraste
  • Imagem importante não tem alternativa textual
  • Modal abre, mas o foco fica perdido na página

Teclado e foco

Não consigo acessar um componente usando Tab

Sintoma típico: o usuário consegue clicar com o mouse, mas não consegue chegar ao componente pelo teclado.

Teclado Componente interativo WCAG 2.1.1 WCAG 2.4.3

Possíveis causas

  • Elemento clicável criado com <div> ou <span>.
  • Uso indevido de tabindex="-1".
  • Componente customizado sem suporte a teclado.
  • Ordem visual diferente da ordem do DOM.

Solução mínima

  • Use <button> para ações e <a href> para navegação.
  • Evite criar controles interativos com elementos neutros.
  • Garanta acionamento por Enter e, quando aplicável, Espaço.
  • Mantenha a ordem do foco coerente com a ordem visual.

Como validar

  • Navegue pela tela usando apenas Tab, Shift+Tab, Enter e Espaço.
  • Confirme se todos os elementos acionáveis recebem foco.
  • Confirme se a sequência do foco é previsível.

O foco visual desaparece ou é difícil de perceber

Sintoma típico: ao navegar por teclado, o usuário não sabe onde está na tela.

Foco visível CSS WCAG 2.4.7 WCAG 2.4.11

Possíveis causas

  • Remoção global de outline no CSS.
  • Indicador de foco com contraste insuficiente.
  • Foco deslocado para elemento fora da área visível.

Solução mínima

  • Nunca use outline: none sem alternativa equivalente.
  • Defina indicador de foco consistente para links, botões, campos e componentes.
  • Garanta contraste e espessura suficientes para o indicador.

Como validar

  • Use Tab do início ao fim da página.
  • Confirme se o foco é visível em todos os componentes.
  • Teste em fundos claros, escuros e imagens.
Exemplo de correção CSS 
a:focus,
button:focus,
input:focus,
select:focus,
textarea:focus {
  outline: 3px solid #000;
  outline-offset: 3px;
}

Semântica e nome acessível

O leitor de tela não informa o tipo ou o nome do controle

Sintoma típico: o leitor de tela anuncia “clicável” ou apenas “botão”, sem explicar a ação.

Nome acessível ARIA WCAG 4.1.2

Possíveis causas

  • Botão composto apenas por ícone sem texto acessível.
  • Componente customizado sem função programática.
  • Uso de ARIA sem atualizar estados, como aria-expanded.

Solução mínima

  • Prefira elementos HTML nativos.
  • Forneça nome acessível com texto visível, aria-label ou aria-labelledby.
  • Atualize estados de componentes expansíveis, selecionáveis ou alternáveis.

Como validar

  • Inspecione a árvore de acessibilidade do navegador.
  • Teste com leitor de tela se o controle informa nome, função e estado.
  • Evite nomes genéricos como “clique aqui”, “ícone” ou “botão”.

Formulários

Campo de formulário é anunciado sem rótulo

Sintoma típico: o leitor de tela anuncia “editável” ou “campo de texto”, mas não informa o que deve ser preenchido.

Formulário Rótulo WCAG 1.3.1 WCAG 3.3.2

Possíveis causas

  • Campo sem <label> associado.
  • Placeholder usado como único rótulo.
  • ID do campo diferente do atributo for do rótulo.

Solução mínima

  • Associe cada campo a um <label>.
  • Use instruções complementares fora do placeholder.
  • Quando houver ajuda ou regra de preenchimento, associe com aria-describedby.

Como validar

  • Clique no texto do rótulo e veja se o foco vai para o campo.
  • Teste com leitor de tela se o campo é anunciado com nome e instrução.
Exemplo de correção HTML 
<label for="email">E-mail</label>
<input id="email" name="email" type="email" autocomplete="email">

Mensagem de erro aparece visualmente, mas não é anunciada

Sintoma típico: após enviar o formulário, a mensagem aparece na tela, mas usuários de leitor de tela não são avisados.

Erro Formulário WCAG 3.3.1 WCAG 4.1.3

Possíveis causas

  • Mensagem inserida dinamicamente sem região viva.
  • Campo inválido sem aria-invalid="true".
  • Erro não associado ao campo correspondente.

Solução mínima

  • Associe a mensagem ao campo com aria-describedby.
  • Marque o campo inválido com aria-invalid="true".
  • Use role="alert" ou aria-live para avisos dinâmicos.

Como validar

  • Envie o formulário vazio usando apenas teclado.
  • Confirme se o erro é anunciado e se o foco permite localizar a correção.

Percepção visual e conteúdo não textual

Texto ou ícone tem baixo contraste

Sintoma típico: a informação existe visualmente, mas é difícil de perceber para pessoas com baixa visão ou em condições adversas de uso.

Contraste Cor WCAG 1.4.3 WCAG 1.4.11

Possíveis causas

  • Texto claro sobre fundo claro ou texto escuro sobre fundo escuro.
  • Ícones, bordas ou estados visuais com contraste insuficiente.
  • Uso de cor como único meio de comunicar estado.

Solução mínima

  • Revise a paleta de cores para texto, ícones e estados de componentes.
  • Use texto, ícone, padrão ou mensagem além da cor.
  • Padronize tokens de cor aprovados para interface.

Como validar

  • Use ferramenta de contraste para texto e componentes.
  • Teste estados: padrão, hover, foco, erro, sucesso e desabilitado.

Imagem importante não tem alternativa textual

Sintoma típico: o leitor de tela ignora uma imagem informativa ou anuncia apenas o nome do arquivo.

Imagem Texto alternativo WCAG 1.1.1

Possíveis causas

  • Imagem informativa sem atributo alt.
  • Texto alternativo genérico, como “imagem” ou “banner”.
  • Imagem decorativa anunciada desnecessariamente.

Solução mínima

  • Descreva a função ou informação da imagem, não apenas sua aparência.
  • Use alt="" para imagens puramente decorativas.
  • Para gráficos complexos, forneça resumo textual e dados em tabela quando necessário.

Como validar

  • Desative imagens ou navegue com leitor de tela.
  • Confirme se a informação essencial continua disponível.

Conteúdo dinâmico e componentes

Modal abre, mas o foco fica perdido na página

Sintoma típico: após abrir uma janela modal, o usuário continua navegando pelo conteúdo que está atrás dela.

Modal Gerenciamento de foco WCAG 2.1.2 WCAG 4.1.2

Possíveis causas

  • Modal sem semântica de diálogo.
  • Foco não movido para o modal na abertura.
  • Conteúdo de fundo ainda disponível para teclado e leitor de tela.
  • Foco não retorna ao botão de origem ao fechar.

Solução mínima

  • Use <dialog> quando possível ou aplique padrão ARIA de diálogo corretamente.
  • Mova o foco para o título ou primeiro controle do modal.
  • Permita fechar com Esc, quando não houver impedimento funcional.
  • Ao fechar, devolva o foco ao elemento que abriu o modal.

Como validar

  • Abra o modal apenas com teclado.
  • Confirme se Tab circula apenas dentro do modal.
  • Feche o modal e confirme se o foco retorna ao ponto inicial.

Modelo de cadastro para novos itens

Em uma versão expandida, cada entrada poderia ser cadastrada em uma planilha, arquivo JSON ou banco de dados e renderizada automaticamente no HTML.

Campos mínimos sugeridos
  • Categoria: teclado, formulário, imagem, contraste, componente dinâmico etc.
  • Sintoma: frase curta do problema observado.
  • Contexto: onde normalmente acontece.
  • Possíveis causas: lista objetiva.
  • Solução mínima: correção prática prioritária.
  • Como validar: teste manual ou automatizado aplicável.
  • Critérios relacionados: WCAG e, se usado, ABNT NBR 17225.
  • Exemplo de código: opcional, quando a correção depende de implementação.
  • Risco se não corrigir: impacto para pessoas usuárias.
Exemplo de estrutura de dados 
{
  "categoria": "Formulários",
  "sintoma": "Campo é anunciado sem rótulo",
  "causas": [
    "Campo sem label associado",
    "Placeholder usado como único rótulo"
  ],
  "solucao_minima": [
    "Associar label ao campo",
    "Associar instruções com aria-describedby"
  ],
  "validacao": [
    "Testar com teclado",
    "Verificar nome acessível na árvore de acessibilidade"
  ],
  "criterios_wcag": ["1.3.1", "3.3.2", "4.1.2"]
}

Observação: este é um protótipo simplificado para avaliar a dinâmica de consulta. Antes de uso institucional, o conteúdo deve passar por revisão técnica, revisão normativa e testes com pessoas desenvolvedoras.
