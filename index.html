Soluções

Não consigo acessar um componente usando Tab

O componente funciona com mouse, mas não entra na sequência de navegação por teclado.

Teclado WCAG 2.1.1 WCAG 2.4.3

Causas prováveis

  • Uso de <div> ou <span> como controle.
  • Uso indevido de tabindex="-1".
  • Componente customizado sem evento de teclado.

Solução mínima

  1. Use <button> para ações.
  2. Use <a href> para navegação.
  3. Garanta acionamento por Enter e, quando aplicável, Espaço.

Como validar

  • Navegue usando apenas Tab e Shift+Tab.
  • Acione os controles pelo teclado.
  • Confirme se a ordem do foco é previsível.
Exemplo de correção 
<button type="button">Abrir filtros</button>

<a href="/relatorios">Ver relatórios</a>

O foco visual desaparece

Ao navegar com Tab, a pessoa não sabe qual elemento está ativo.

Foco visível WCAG 2.4.7 WCAG 2.4.11

Causas prováveis

  • Remoção de outline no CSS.
  • Indicador de foco com contraste insuficiente.
  • Foco enviado para área fora da tela.

Solução mínima

  1. Não remova outline sem alternativa equivalente.
  2. Padronize foco para links, botões e campos.
  3. Use contraste, espessura e afastamento suficientes.

Como validar

  • Percorra a página usando Tab.
  • Teste em fundos claros, escuros e imagens.
  • Confirme se todos os componentes têm foco visível.
Exemplo de correção 
a:focus,
button:focus,
input:focus {
  outline: 3px solid #000;
  outline-offset: 3px;
}

Campo de formulário é anunciado sem rótulo

O leitor de tela informa apenas “campo de texto” ou “editável”, sem indicar o que deve ser preenchido.

Formulário WCAG 1.3.1 WCAG 3.3.2 WCAG 4.1.2

Causas prováveis

  • Campo sem <label> associado.
  • Placeholder usado como único rótulo.
  • for do rótulo diferente do id do campo.

Solução mínima

  1. Associe cada campo a um rótulo visível.
  2. Use placeholder apenas como apoio.
  3. Associe instruções com aria-describedby, quando necessário.

Como validar

  • Clique no rótulo e veja se o foco vai para o campo.
  • Inspecione o nome acessível do campo.
  • Teste com leitor de tela.
Exemplo de correção 
<label for="email">E-mail</label>
<input id="email" name="email" type="email" autocomplete="email">

Erro aparece na tela, mas não é anunciado

Depois do envio, a mensagem de erro aparece visualmente, mas não chega ao leitor de tela.

Erro WCAG 3.3.1 WCAG 3.3.3 WCAG 4.1.3

Causas prováveis

  • Mensagem dinâmica sem região viva.
  • Campo inválido sem aria-invalid.
  • Erro não associado ao campo.

Solução mínima

  1. Associe o erro ao campo com aria-describedby.
  2. Use aria-invalid="true" no campo inválido.
  3. Use role="alert" ou aria-live para avisos dinâmicos.

Como validar

  • Envie o formulário vazio usando teclado.
  • Confirme se o erro é anunciado.
  • Confirme se a pessoa consegue localizar o campo com erro.

Texto ou ícone tem baixo contraste

A informação existe visualmente, mas é difícil de perceber.

Contraste WCAG 1.4.3 WCAG 1.4.11

Causas prováveis

  • Texto claro sobre fundo claro.
  • Ícones e bordas com baixo contraste.
  • Cor usada como único meio de indicar estado.

Solução mínima

  1. Revise cores de texto, ícones, bordas e estados.
  2. Use texto, ícone ou padrão visual além da cor.
  3. Padronize tokens de cor aprovados.

Como validar

  • Meça contraste de texto e componentes.
  • Teste estados: foco, erro, sucesso, hover e desabilitado.
  • Confira se a informação não depende apenas da cor.

Imagem importante não tem alternativa textual

O leitor de tela ignora uma imagem informativa ou anuncia apenas o nome do arquivo.

Imagem WCAG 1.1.1

Causas prováveis

  • Imagem informativa sem atributo alt.
  • Texto alternativo genérico.
  • Imagem decorativa anunciada desnecessariamente.

Solução mínima

  1. Descreva a função ou informação da imagem.
  2. Use alt="" para imagens decorativas.
  3. Para gráficos complexos, ofereça resumo e dados em tabela.

Como validar

  • Navegue com leitor de tela.
  • Confirme se a informação essencial continua disponível.
  • Verifique se imagens decorativas não geram ruído.

Modal abre, mas o foco fica perdido na página

Depois de abrir o modal, a navegação por teclado continua passando pelo conteúdo que está atrás dele.

Modal WCAG 2.1.2 WCAG 2.4.3 WCAG 4.1.2

Causas prováveis

  • Modal sem semântica de diálogo.
  • Foco não movido para o modal na abertura.
  • Conteúdo de fundo ainda navegável.
  • Foco não retorna ao acionador ao fechar.

Solução mínima

  1. Use <dialog> quando possível.
  2. Mova o foco para o modal ao abrir.
  3. Mantenha Tab dentro do modal.
  4. Devolva o foco ao botão de origem ao fechar.

Como validar

  • Abra o modal usando teclado.
  • Confirme se Tab não vai para o conteúdo de fundo.
  • Feche com Esc, quando aplicável, e confira o retorno do foco.
